# LAO v0.9 아키텍처 문서

이 문서는 시각적 마인드맵 표현에서 **문서 기반 가이드형 기획 작업 공간(workspace)**으로의 전환에 중점을 둔 **LAO v0.9**의 아키텍처 원칙, 파일 시스템 구조 및 운영 워크플로우를 설명합니다.

---

## 1. 디자인 철학

v0.9의 주된 목적은 개발자의 **인지적 피로도를 최소화**하면서 **코드와 기획 명세(spec) 간의 차이를 방지**하는 것입니다.

이를 달성하기 위해 우리는 수동적이고 노드 단위의 마인드맵 그래프 에디터에서 **감독관 제어 게이트가 포함된 자동화 오토파일럿**으로 전환했습니다:
1. **오토파일럿 도입 및 개념 분기 (v0.9.2)**: 개발자가 대략적인 아이디어를 입력하면 AI 에이전트들이 3가지 고유한 제품 제안(Fast MVP, 구조적 확장, 차별화된 실험)과 함께 디렉터 추천안을 생성합니다. 개발자는 옵션을 검토하고 사용자 정의 조정을 입력한 다음 경로를 선택하여 코어 명세와 기능 세트를 생성(sprout)합니다.
2. **의사결정 분기 (개입)**: 그래프를 미세하게 관리하는 대신 개발자는 감독관 역할을 하여 자동 생성을 조정하는 상위 레벨의 옵션 카드(예: SQLite vs PostgreSQL)를 승인합니다.
3. **가드레일 (골든 룰)**: 의사결정은 모든 에이전트 컨텍스트에 주입되는 엄격한 기술 사전 설정(기술 가드레일)에 의해 제한됩니다.
4. **코드 잠금 단계 게이트 (Lock-to-Code Phase Gate)**: AI가 승인된 요구 사항에서 벗어나는 것을 방지하기 위해 코드 생성이 시작되기 전에 기획 명세서가 잠금 상태로 설정됩니다.

## 2. 디렉토리 및 저장 구조

모든 프로젝트 구성, 명세서, 의사결정 및 채팅 로그는 `.lao/` 디렉토리에 깔끔하고 버전 관리가 가능한 파일로 저장됩니다:

```text
.lao/
├── lao.config.json           # 프로젝트 설정, 골든 룰, 단계(phase) 및 프로바이더 구성
├── proposals.json            # 초기 아이디어에서 생성된 개념 제안서 (Option A, B, C)
├── criteria.md               # 채택된 의사결정 및 논거의 시간순 로그
├── spec_compiled.md          # 코어 및 활성 기능 명세서의 결합본
├── messages.json             # 다중 에이전트 사무실과의 채팅 히스토리
├── mockup.html               # MockupGenerator에 의해 생성된 완전 인터랙티브 UI 목업
├── task.md                   # 개발 단계를 위해 명세서로부터 생성된 체크리스트 태스크
├── specs/
│   ├── core_spec.md          # 기본 아키텍처 및 기술 스택 명세
│   └── features/
│       ├── [feature_1].md    # 활성 기능 명세 (YAML 스타일 프론트매터 포함)
│       └── [feature_2].md
└── decisions/
    ├── [decision_1].json     # Optionizer가 제안한 대기 중이거나 결정된 의사결정 카드
    └── [decision_2].json
```

### 명세서 프론트매터(Frontmatter) 포맷
기능 명세서는 YAML 형식의 헤더가 있는 표준 마크다운 파일입니다:
```markdown
---
id: auth_system
title: User Authentication System
status: active
createdAt: 2026-05-26T06:50:00Z
updatedAt: 2026-05-26T06:51:00Z
---
# Requirement Details
...
```

## 3. 다중 에이전트 협업 루프

작업 공간은 명세서에 대해 협업하는 특화된 AI 에이전트 페르소나들을 활용합니다:

```mermaid
graph TD
    User([대략적인 아이디어 입력]) --> Director[디렉터 에이전트]
    Director --> Specifier[스펙 파서/작성 에이전트 (Specifier)]
    Director --> Optionizer[옵션 분석 에이전트 (Optionizer)]
    Director --> GapDetector[갭 탐지 에이전트 (Gap Detector)]
    Director --> MockupGen[목업 생성 에이전트 (Mockup Generator)]
    
    Specifier -->|1. 명세 생성/분기| Core[core_spec.md]
    Specifier -->|1. 명세 생성/분기| Feat[features/*.md]
    
    Feat --> Optionizer
    Optionizer -->|2. 대안 제안| Decisions[decisions/*.json]
    
    Decisions -->|3. 선택 승인| Criteria[criteria.md]
    
    Core & Feat --> GapDetector
    GapDetector -->|4. 누락 검출| Gaps[갭 감사 보고서]
    
    Core & Feat -->|명세 컴파일| SpecCompiled[spec_compiled.md]
    SpecCompiled & User --> MockupGen
    MockupGen -->|5. UI 목업 생성| Mockup[mockup.html]
```

### 골든 룰 강제 적용
`Specifier` 또는 `Optionizer`가 호출될 때 프롬프트 템플릿은 `lao.config.json`에 정의된 `goldenRules`를 동적으로 주입합니다. 이를 통해 모든 제안이 개발자가 원하는 기술 사양(예: 'Docker나 PostgreSQL 없이 SQLite만 사용' 등)에 맞게 조정됩니다.

### 목업 생성 에이전트 (Mockup Generator Agent)
기획 명세서가 컴파일되면 `Mockup Generator`가 컴파일된 마크다운 명세서와 사용자 피드백을 읽어 `.lao/mockup.html`을 생성하거나 업데이트합니다. 이 목업은 바닐라 자바스크립트로 작성된 인터랙티브하고 미려한 스타일링을 갖추고 있어, 기획 단계에서 기능적인 UI 프로토타입 역할을 수행합니다.

## 4. 단계 잠금 및 탭 제어 (기획 -> 개발)

코드와 기획 명세서 간의 차이를 방지하기 위해 시스템은 엄격한 단계 잠금(Phase-Lock) 프로토콜을 구현합니다:
* **기획 단계 (Planning Phase)**: 기획 명세서를 완전히 수정할 수 있습니다(Web UI에서 직접 인라인 편집을 하거나 `.lao/specs/` 하위의 마크다운 파일을 편집). 옵션 카드를 확인하여 결정할 수 있습니다. 인터랙티브 어플리케이션 목업(`mockup.html`)이 생성되어 **Live Preview** 탭에서 확인할 수 있습니다. 그러나 개발자가 기획 확정 전에 코드 생성이나 개발자 도구를 실행하는 것을 방지하기 위해 실행 기능(DevLoop Console) 및 최종 라이브 앱 프리뷰(Live App Preview)는 차단 및 잠금 상태가 됩니다.
* **개발 단계 (Development Phase)**: 명세서가 읽기 전용 상태로 잠깁니다. 구현을 가이드하기 위해 태스크 체크리스트(`.lao/task.md`)가 컴파일됩니다. 코딩 에이전트가 `spec_compiled.md`를 읽고 코드 생성을 시작합니다. **DevLoop Console**과 **Live Preview** 탭이 잠금 해제되어, 개발 명령(빌드, 실행, 검증)을 실행하고 출력을 실시간으로 스트리밍하는 동시에 실제 실행 중인 애플리케이션을 렌더링할 수 있습니다. 명세서를 수정하려면 기획 단계로 다시 전환해야 하며, 이 경우 활성화된 코딩 루프는 중지됩니다.

## 5. 명세서 품질 관리 및 프로세스 실행 (v0.9.3)

### 5.1 기획 하네스 (Planning Harness) 및 프로그램식 린터
완성도 높고 실제 구현 가능한 사양을 유지하기 위해, 명세서를 수정하는 모든 에이전트 출력물은 `PlanningHarness` 검증 레이어를 거칩니다.
* **코어 명세서 검증**: 기능의 무분별한 팽창(Feature Creep)을 방지하기 위해 `## Out of Scope (Non-Goals)` 섹션이 반드시 정의되어야 합니다.
* **기능 명세서 검증**: `## User Story`("As a... I want to... So that..." 형식 사용) 및 `Given-When-Then` 구문으로 구조화된 `## Acceptance Criteria`가 반드시 포함되어야 합니다.
* **RULES.md 연동**: 기술 스택의 모순(예: 프로젝트 규칙이 Tailwind를 강제하는데 Bootstrap을 제안하는 경우 등)을 잡아내기 위해, 제안된 모든 사양을 프로젝트 루트의 `RULES.md` 가이드라인과 정적으로 비교하는 린팅 단계가 가동됩니다.
* **CLI 환경 진단**: 하네스는 `runDiagnostics()`를 통해 로컬 CLI 환경(Claude, Gemini, Codex, Cursor, AGY) 기동 경로의 유효성 및 API Key 연결 자격 증명을 사전 체크합니다.

### 5.2 자가 교정 루프 및 사용자 개입(HITL) 수동 반영 (Force Commit)
* **자가 교정 루프**: Specifier 에이전트가 기획을 업데이트할 때 검증 실패 에러가 발생하면, 시스템은 최대 3회까지 에러 피드백을 모델 프롬프트 버퍼에 전달하여 자가 교정을 시도합니다.
* **수동 강제 반영 (Force Commit - HITL Bypass)**: 3회의 자동 교정 시도가 모두 실패하면 시스템은 자동 반영을 일시 중단합니다. React Web UI 화면에 하네스 정적 검증 에러 로그를 상세히 출력하고, 개발자가 원한다면 기획 하네스 필터를 무시하고 작성한 내용 그대로 커밋할 수 있도록 **[Force Commit]** 버튼을 제공하여 완벽한 사용자 통제권을 보장합니다.

### 5.3 동시성 제어 및 프로세스 스케줄링 (`SpawnQueueManager`)
로컬 시스템 자원(특히 macOS 환경)을 보호하고 매끄러운 UI 업데이트 흐름을 제공하기 위해 백엔드에 전용 프로세스 스케줄러를 도입했습니다:
* **동시성 실행 한도**: 동시 기동 가능한 프로세스 개수를 최대 `maxConcurrency = 2`로 제한하여 CPU 고갈 및 SQLite 데이터베이스의 락 이슈(`SQLITE_BUSY`)를 미연에 방지합니다.
* **우선순위 스케줄링**: 작업 큐(Spawn Queue)에 진입한 태스크들은 중요도에 따라 우선순위가 정해지며, 실시간 채팅 스트리밍이나 디렉터의 동적 라우팅 같은 `high` 우선순위 작업이 우선 배치됩니다.
* **목업 생성 태스크 데듀플리케이션(중복 제거)**: 새로운 목업 생성 요청이 들어왔을 때, 큐에 대기 중이던 이전 목업 생성 요청이 있다면 최신 변경사항을 신속히 반영하기 위해 이전 요청을 대기열에서 제거(Eviction)하고 반려합니다.
* **오작동 타임아웃 보호**: 개별 프로세스는 최대 90초의 라이프타임 제한을 받습니다. 이 시간을 초과하여 멈춰 있는 태스크는 스케줄러가 `SIGKILL` 시그널을 송신하여 좀비 프로세스가 되는 것을 강제 차단합니다.

### 5.4 성능 및 안정성 최적화 요소
* **zsh 기동 오버헤드 단축**: macOS의 로그인 셸 프로필 로딩(`zsh -lc`) 방식을 배제하고 논로그인 셸(`zsh -c`) 방식을 차용하여 프로세스 실행 지연 시간을 약 10ms 단위 수준으로 단축했습니다.
* **ARG_MAX 셸 오버헤드 우회**: 프롬프트의 분량이 극도로 길어지는 경우 발생하는 Unix 셸의 `E2BIG` (Argument list too long) 에러를 방지하기 위해, 프롬프트를 임시 파일에 기록한 후 파일 입력 리다이렉션(`<`) 형태로 실행 매개변수를 우회 전달합니다.
* **지수 백오프 재시도**: 일시적인 API 속도 제한(`rate_limited`)이나 SQLite 데이터베이스 락 충돌 시, 지수 백오프 알고리즘에 기초하여 자동 재시도를 수행해 신뢰성을 유지합니다.
* **컨텍스트 버제팅 (Context Budgeting)**: 라우팅되는 에이전트 페르소나에 최적화하여 연관 없는 사양 섹션은 잘라내고 핵심 정보만 요약 전달함으로써 불필요한 토큰 낭비 및 컨텍스트 한계 붕괴를 방지합니다.
